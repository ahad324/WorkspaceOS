# Context

WorkspaceOS is intended to become a long-lived infrastructure project.

The Core Runtime is responsible for:

• Workspace Management
• Security
• Indexing
• Context Generation
• Repository Graph
• Scheduling
• Plugin Runtime
• MCP Runtime
• Tunnel Management

This runtime must be:

• Extremely fast
• Memory efficient
• Highly concurrent
• Reliable
• Secure
• Cross-platform
• Maintainable for many years

Selecting the correct implementation language is therefore the most
important architectural decision in the project.

--------------------------------------------------------------------

# Requirements

The Core Language must provide:

✓ Native performance

✓ Small memory footprint

✓ Excellent concurrency

✓ Strong type safety

✓ Excellent tooling

✓ Cross-platform compilation

✓ Mature ecosystem

✓ Long-term stability

✓ Low runtime overhead

✓ Strong security guarantees

# Decision

WorkspaceOS SHALL use Rust as its Core implementation language.

--------------------------------------------------------------------

# Rationale

Rust provides the strongest combination of:

Performance

Memory efficiency

Safety

Concurrency

Long-term maintainability

Cross-platform support

Rust aligns directly with WorkspaceOS priorities.

--------------------------------------------------------------------

# Consequences

Positive

✓ Native performance

✓ Low idle memory

✓ Low CPU usage

✓ Memory safety

✓ Thread safety

✓ Excellent async runtime

✓ Strong ecosystem

✓ Excellent integration with Tauri

Negative

• Higher learning curve

• Slightly slower initial development

These tradeoffs are acceptable.

--------------------------------------------------------------------

# Engineering Rules

The Core SHALL:

Use safe Rust by default.

Avoid unnecessary unsafe code.

Benchmark optimizations.

Prefer ownership over shared mutable state.

Use async only where appropriate.

Prefer compile-time guarantees over runtime validation whenever
practical.

--------------------------------------------------------------------

# Future Review

This decision SHALL only be reconsidered if:

A future language demonstrates objectively superior characteristics
across performance, safety, tooling, maintainability and ecosystem.

Such a change would require:

• New benchmarks
• Updated ADR
• Architecture review
• Migration strategy

Decision: ACCEPTED
